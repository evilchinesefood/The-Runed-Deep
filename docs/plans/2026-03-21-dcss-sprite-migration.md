# DCSS Sprite Migration Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Replace all game sprites with Dungeon Crawl Stone Soup (DCSS) 32x32 tiles for a cohesive visual upgrade.

**Architecture:** Download the DCSS tileset (CC0, 6000+ sprites at 32x32), build a Python sprite-packing tool that maps each game CSS class to a DCSS source tile, generates new sprite sheets and CSS files, then swap them in. No game logic changes — only asset files and CSS coordinates change.

**Tech Stack:** Python 3 (Pillow for image processing), existing CSS sprite sheet format, Vite build pipeline.

## Task 6: Build the Sprite Packer Tool

**Files:**
- Create: `tools/SpritePacker.py`

This script reads a mapping JSON, loads each DCSS source PNG, packs them into a single sprite atlas (PNG), and generates the corresponding CSS file.

- [ ] **Step 1: Install Pillow**

```bash
pip3 install Pillow
```

- [ ] **Step 2: Write SpritePacker.py**

```python
#!/usr/bin/env python3
"""
Reads a mapping JSON { cssClass: dcssPath } and produces:
  - A packed sprite atlas PNG (tiles arranged in a grid, 32x32 each)
  - A CSS file mapping each class to its position in the atlas
"""
import json, sys, os, math
from PIL import Image

def pack(mapping_file, atlas_out, css_out, png_ref_path, base_class):
    with open(mapping_file) as f:
        mapping = json.load(f)

    classes = list(mapping.keys())
    count = len(classes)
    cols = min(14, count)  # max 14 per row (448px wide)
    rows = math.ceil(count / cols)

    atlas = Image.new('RGBA', (cols * 32, rows * 32), (0, 0, 0, 0))
    css_lines = [f'.{base_class} {{\n    height: 32px;\n    width: 32px;\n    display: inline-block;\n}}\n']

    for i, cls in enumerate(classes):
        src_path = os.path.join('dcss-tiles', mapping[cls])
        col = i % cols
        row = i // cols
        x = col * 32
        y = row * 32

        if os.path.exists(src_path):
            img = Image.open(src_path).convert('RGBA').resize((32, 32), Image.NEAREST)
            atlas.paste(img, (x, y))
        else:
            print(f'WARNING: missing {src_path} for .{cls}')

        css_lines.append(f'.{cls} {{\n    background: url(\'{png_ref_path}\') -{x}px -{y}px;\n}}')

    atlas.save(atlas_out)
    with open(css_out, 'w') as f:
        f.write('\n'.join(css_lines) + '\n')

    print(f'Packed {count} sprites into {atlas_out} ({cols}x{rows} grid)')
    print(f'Generated {css_out}')

if __name__ == '__main__':
    # Usage: python3 SpritePacker.py <mapping.json> <atlas.png> <output.css> <css-url-path> <base-class>
    pack(sys.argv[1], sys.argv[2], sys.argv[3], sys.argv[4], sys.argv[5])
```

## Risk Notes

1. **Mapping quality is the bottleneck.** The tooling is simple — the hard part is choosing the right DCSS tile for each of 314 sprites. Plan 2-3 hours for manual curation across Tasks 2-5.

2. **Some DCSS tiles may not be 32x32.** The packer resizes with `Image.NEAREST` to preserve pixel art quality. If a source tile is larger (e.g., 64x64 boss sprite), it will be downscaled.

3. **Buildings are NOT replaced.** The multi-tile building system uses hardcoded pixel dimensions that don't map to 32x32 grid tiles. Replacing buildings is a separate, larger task.

4. **Spell CSS has a typo alias.** Both `.lightnig-bolt-spell` and `.lightning-bolt-spell` must be in the mapping. The packer handles this — just include both entries pointing to the same DCSS source.

5. **The `inventory.css` paperdoll is NOT replaced.** It's a custom illustration, not a tile grid. The equipment-dude works well and doesn't need a DCSS equivalent.

6. **Town tile tinting.** The game applies CSS `background-blend-mode: multiply` with a color tint for Fortress (blue) and Castle (brown-red) dungeon walls. This will still work with DCSS wall tiles — the tint is applied at render time, not baked into the sprite.
